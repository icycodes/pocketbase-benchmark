## Update Rule Explained

The rule uses PocketBase's **back-relation syntax** (`<collection>_via_<field>`):

- `document_edit_permissions_via_document` — resolves to all `document_edit_permissions` rows whose `document` field equals the id of the document being updated
- `?=` — "at least one" operator, checks if any of those rows has `grantee` matching `@request.auth.id`

The migration is structured in 3 steps to avoid a chicken-and-egg validation problem: `documents` is saved first (without update rule), then `document_edit_permissions` is saved, and finally `documents` is patched with the back-relation update rule.

## Key Implementation Detail

The migration uses a 3-step approach because PocketBase validates the `updateRule` at save time — if `document_edit_permissions` doesn't exist yet, the back-relation in the rule would fail validation. Saving `documents` without the rule first, creating `document_edit_permissions`, then patching the rule on `documents` resolves this dependency ordering.
