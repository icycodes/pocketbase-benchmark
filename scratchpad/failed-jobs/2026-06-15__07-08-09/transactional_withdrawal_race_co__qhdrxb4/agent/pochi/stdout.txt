### Key Implementation Details

1. **PocketBase v0.23+ Event Hooks**:
   We utilized the `OnRecordCreateRequest` hook on the `withdrawals` collection to intercept withdrawal creations.
   
2. **Atomic Balance Deduction & SQLite Concurrency**:
   To prevent race conditions and lost updates without relying on heavy lock configurations, we implemented an **atomic `UPDATE` statement** executed within a database transaction:
   ```sql
   UPDATE wallets SET balance = balance - :amount WHERE id = :id AND balance >= :amount
   ```
   * **Why this is safe**: SQLite serializes write operations. Executing this atomic statement as the first database operation inside the transaction locks the row/database for writing immediately, avoiding any dirty reads or race conditions.
   * **Atomic Validation**: If the balance is insufficient, the `UPDATE` affects `0` rows. We then check if the wallet exists to return a precise `404 Not Found` ("Wallet not found") or `400 Bad Request` ("Insufficient funds").

3. **Transaction Safety**:
   We wrapped the hook in `e.App.RunInTransaction(func(txApp core.App) error { ... })` and set `e.App = txApp` before calling `e.Next()`. This ensures that the record creation itself runs within the same database transaction. If `e.Next()` or any validation fails, the entire transaction (including the balance deduction) is automatically rolled back.

4. **Bootstrapping Database Collections**:
   We implemented automatic schema verification on startup within `OnServe()`. If the `wallets` or `withdrawals` collections do not exist, they are created programmatically with the correct schemas, rules, and relations.
