### `main.go`
Implements the transactional withdrawal hook with two hooks bound:

#### `OnRecordCreateRequest("withdrawals")` — main hook
```
1. Validate wallet_id and amount present / positive
2. Call app.RunInTransaction(func(txApp) {
     UPDATE wallets
       SET balance = balance - :amount
     WHERE id = :wallet_id AND balance >= :amount
   })
3. If 0 rows affected → 400 "insufficient funds" (transaction rolls back)
4. If UPDATE succeeded → call e.Next() to persist the withdrawal record
```

#### `OnRecordAfterCreateError("withdrawals")` — compensating hook
If the withdrawal record fails to save after the balance was already deducted, a best-effort `UPDATE wallets SET balance = balance + :amount` is executed to restore the balance.

---

### How race conditions are prevented

| Layer | Mechanism |
|---|---|
| **Single writer connection** | `RunInTransaction` routes to `NonconcurrentDB` which is configured with `SetMaxOpenConns(1)` — all write transactions are serialised through one SQLite connection |
| **Atomic check-and-deduct** | The `UPDATE … WHERE balance >= amount` is a single SQL statement; there is no separate SELECT → UPDATE gap that could be raced |
| **Busy timeout** | SQLite WAL is configured with `busy_timeout(10000)` — goroutines waiting for the connection retry for up to 10 s |
